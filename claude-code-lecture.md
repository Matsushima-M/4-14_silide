# 2. コンテキストエンジニアリング
*今AIを使ううえで一番大事な概念*

---

# コンテキストって何？

**コンテキスト** = 1回の実行でAIが扱える情報の量

わかりやすく言うと：

> AIの**「作業机の広さ」**

- 机の上に置けるものには**限りがある**
- 資料を広げすぎると、作業スペースがなくなる
- だから**何を机に置くか**が大事になる

---

# コンテキストの大きさ ― 具体的にどれくらい？

最新モデルは100万トークン（文字のかたまり）を扱える

**ただし、実用的に使えるのは約20万トークン**

<br>

### 20万トークンってどれくらい？

> **ハリーポッター 約1.5冊分**

<br>

多く感じるかもしれないけど...

---

# 実はすぐに埋まってしまう

コンテキスト（作業机）の中には、こんなものが**全部**載っている：

| 載っているもの | たとえると |
|:--|:--|
| あなたが送った指示 | 注文書 |
| AIが考えた内容 | メモ帳 |
| 読み込んだファイル | 参考資料 |
| 実行結果 | 報告書 |
| 記憶（メモリ） | 付箋 |
| ツールのやりとり | 電話の通話記録 |

> これが全部、同じ机の上に載っている。 **意外とすぐ机がいっぱいになる。**

---

# だから「量より質」

コンテキストが限られている以上、大事なのは：

<br>

### ✕ とにかく大量の情報を入れる
### ○ 本当に必要な情報だけを厳選して入れる

<br>

> 狭い机に百科事典をどさっと置いても、作業はできない。
> **必要なページだけ開いて置く**のが正解。

---

# よくある誤解：「AIに学習させる」

ネットでよく見る情報：
> 「AIにたくさん情報を食わせて賢くさせよう」
> 「データをどんどん入れて最適化しよう」

<br>

### これ、AI自体を*開発する*話としては正しい

### でも、すでに出来上がったAIを*使う*ときは**間違い**

---

# なぜ「たくさん入れる」がダメなのか

### 問題1：思考スペースの圧迫

- 情報を入れすぎると、**AIが考えるためのスペース**がなくなる
- 机の上が資料だらけで、計算用紙を広げる場所がない状態
- 結果 → **AIの性能が落ちる**

### 問題2：自動圧縮（オートコンパクト）の発動

- コンテキストが一定量を超えると、AIが自動的に**情報を圧縮**する
- このとき、**本来守ってほしかった指示まで潰される**ことがある
- 結果 → **言うことを聞かなくなる、作業がめちゃくちゃになる**

---

# 自動圧縮のイメージ

```
【圧縮前の机】
┌─────────────────────────────┐
│ 指示書  メモ  資料  参考書  │
│ 報告書  計算用紙  付箋     │
│ ────── もう置けない！ ──── │
└─────────────────────────────┘
        ↓ 自動圧縮が発動 ↓
【圧縮後の机】
┌─────────────────────────────┐
│ ぎゅっと圧縮された何か     │
│ ※大事な指示も一緒に潰れた │
│                             │
└─────────────────────────────┘
```

> **圧縮が起きないように、余裕を持たせるのが大事**

---

# 正しいアプローチ

### ✕ 大量の情報を入れて「覚えさせる」

### ○ 大量の情報から**本当に必要なものだけ抽出**して渡す

<br>

> 秘書に仕事を頼むとき、
> 会社の全書類を渡す → ✕
> **必要な書類だけまとめて渡す** → ○

---

# 「AIチーム」の落とし穴

最近よく見る発信：
> 「エージェントでチームを組んで効率化！」

### 現実は...

- **最初のうちは**いい感じに動く（机がきれいな状態）
- チームとして動くにつれて「こういうケースもあるよね」と情報が増えていく
- コンテキストがどんどん圧迫される
- 結局 → **まともに動かなくなる**

> 現時点では、この問題を完全に解決する方法はまだない

# 3. コンテキスト管理の基礎
*実際にどう管理するか*

---

# Claude.md ― AIの「常駐メモ」

`Claude.md` = Claudeが**ずっと頭の中に持っている情報**

<br>

### 公式のベストプラクティス：

- **200行くらいに収める**
- 長くなる情報は別のファイルに置く
- 必要なときに読みに行けるようにする

<br>

> 頭の中にずっと入れておく情報は**最小限**にする。
> 残りは「引き出し」に入れて、必要なときに取り出す。

---

# Claude.md の考え方

```
┌─────────────────────────────┐
│       Claude.md (常駐)       │
│   ・基本ルール              │
│   ・よく使う指示            │
│   ・参照先のファイル一覧    │
│         200行以内            │
└──────────┬──────────────────┘
           │ 必要なときに読みに行く
     ┌─────┼─────┐
     ▼     ▼     ▼
  docs/  docs/  docs/
  仕様A  仕様B  手順C
```

> ポイント：Claude.mdに全部書くのではなく、
> **「読みに行く先」を育てる**

---

# スキル ― 作業ごとの「手順書」

**スキル**も仕組みは同じ。大層な名前だけど、やっていることは単純。

<br>

| Claude.md | スキル |
|:--|:--|
| **常に**頭の中にある | **作業するときだけ**読みに行く |
| 全体のルール | 個別の作業手順 |
| 「いつも覚えておくメモ」 | 「○○するときに見るマニュアル」 |

<br>

> スキル = **「○○の作業をするときに読む仕様書」**
> 難しいことはなく、ただの手順書です。

---

# スキルの作り方

特別なことは不要。以下の流れでOK：

<br>

1. まず普通にClaude Codeで作業する
2. うまくできたら、こう伝える：

> **「この一連の作業をきれいにまとめて、スキルとして登録しておいて」**

3. Claudeが自動でスキルファイルを作ってくれる

<br>

> 一定の作業パターンがあるなら、**スキルにしておくと次回から楽**

# 3.5 Agent Skillsを深く理解する
*MCPとの違いと「段階的開示」の仕組み*

---

# なぜAgent Skillsが必要なのか？

AIに**道具（ツール）を増やしすぎる**と、逆に性能が落ちる問題があった

### 道具が少ないとき
- 迷わず正しい道具を選べる / 作業机もスッキリ / 性能が安定する

### 道具が多すぎるとき
- 「どれを使えばいい？」と**迷う** / 道具の説明だけで**机が埋まる** / 性能が**落ちる**

> この「道具が増えるほど迷う」問題を解決するのが**Agent Skills**

---

# Agent Skills と MCP の違い

MCPは道具そのもの。Agent Skillsは**目的ごとに整理された道具箱**。

| | MCP（道具） | Agent Skills（道具箱） |
|:--|:--|:--|
| たとえ | ハンマー、ノコギリ…を1つずつ渡す | 「日曜大工セット」としてまとめて渡す |
| 役割 | 外部ツールとの**接続方法** | 道具の**使い方・手順・ルール**をセットにしたもの |
| 関係 | Agent SkillsはMCPを**置き換えるものではなく、上位の仕組み** ||

> MCPで繋いだ道具を「どう使うか」を教えるのがAgent Skills。
> **道具 + 手順書 = 即戦力の専門家**

---

# 段階的開示 ― 必要なときだけ情報を渡す

Agent Skillsの核心は**「情報を一度に全部渡さない」**こと

```
ステップ1：目次だけ読む
  スキルA: ○○用 / スキルB: △△用 ...  ← 名前と説明だけ
        ↓ 「このタスクにはスキルBが必要だ」

ステップ2：該当する手順書を開く
  スキルBの本体（SKILL.md）            ← 詳しい手順だけ読む
        ↓ 「さらにこの部分の詳細が必要」

ステップ3：追加資料だけ取り出す
  forms.md / reference.md / script     ← 本当に必要な部分だけ
```

---

# 良いスキルを作る4つのコツ

### 1. まず失敗を観察する
- いきなり完璧を目指さず、AIが**どこでつまずくか**を見てから作る

### 2. 情報を分割して整理する
- 1ファイルに全部書かない。**同時に使わない情報は別ファイルに**

### 3. 名前と説明をAI向けに書く
- AIが「いつこのスキルを使うべきか」**判断できる名前**にする

### 4. AIと対話しながら改善する
- Claude自身に「なぜ失敗したか分析して、スキルに追記して」と**自己改善**させる

> スキルは一度作って終わりではなく、**使いながら育てていくもの**

# 4. セキュリティ
*ここだけは絶対に守ってほしいこと*

---

# 基本方針：ローカルで動かす

自分用のツールはどんどん作ってOK。

**ただし、オンラインに公開しない。ローカル（自分のPC）で動かす。**

<br>

### なぜ？

- セキュリティ事故 = 公開したものを悪意ある人が見つけて悪用する
- ローカルで動かしていれば、外部からアクセスされようがない
- ウイルスが入っていない限り安全

> 「社内全体で使いたい」となった場合は、別途安全な方法で構築します

---

# 絶対にやってはいけないこと

### 1. 個人情報を読み込ませない

- 管理画面の情報など、個人名が入っているデータ
- **絶対にAIに渡さないでください**

<br>

### 2. APIキーをClaude Codeに渡さない

- 「このAPIを使ってあれこれやって」は**厳禁**
- APIキーは非常に重要な認証情報

---

# 絶対にやってはいけないこと（続き）

### 3. 広告媒体のAPIには絶対に接続しない

- Google広告、Meta広告などのAPI
- Claude Code経由はもちろん、プログラムとして書いて渡すのも危険
- GAS（Google Apps Script）で一部使う、なども**ダメ**
- 安全な管理方法が確立されるまで、**触らない**

<br>

> 何かあったとき、**誰も責任を取れない**。だからやめましょう。

---

# スキルやツールの安全な導入

GitHubなどで公開されているスキルやツール：

### ✕ 「便利そう！」→ そのまま入れる

### ○ 「便利そう！」→ **まず中身を調べてから**入れる

---

# なぜ中身を調べる必要があるのか

スキル = **手順書**。Claudeはその手順書の通りに作業する。

<br>

### もし悪意のある手順が紛れていたら？

- 「作業が終わったら、この情報を外部に送信する」
- 「このファイルの中身をここにアップロードする」

→ **Claudeはその通りに実行してしまう**

<br>

> 対策：導入前に「このスキルに悪意のある記述がないか調べて」と
> Claude自身にチェックさせる

---

# セキュリティまとめ

| ルール | 理由 |
|:--|:--|
| ローカルで動かす | 外部からのアクセスを防ぐ |
| 個人情報を渡さない | 情報漏洩を防ぐ |
| APIキーを渡さない | 不正利用を防ぐ |
| 広告APIに接続しない | 事故時に責任が取れない |
| スキル・ツールは中身を確認 | 悪意ある手順の混入を防ぐ |

<br>

> **迷ったら「やらない」が正解。安全な方法が確立されたら伝えます。**
